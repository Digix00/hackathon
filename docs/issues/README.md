# Issues 管理

このフォルダは、プロジェクトの技術的な課題や改善項目を管理するためのものです。

## Issue ファイルの命名規則

```
{番号}-{簡潔な説明}.md
```

例:
- `001-long-text-truncation.md`
- `002-network-error-handling.md`
- `003-performance-optimization.md`

## Issue テンプレート

新しいissueを作成する際は、以下のテンプレートを使用してください。

```markdown
# Issue #{番号}: {タイトル}

## 概要
問題や改善項目の簡潔な説明

## 発生する可能性のある問題
具体的な問題点をリスト化

## 対応方針
実装方法や解決策の提案

## 実装スケジュール
- [ ] タスク1
- [ ] タスク2
- [ ] タスク3

## テスト項目
検証すべき項目のリスト

## 参考資料
関連するドキュメントやリンク

## 関連Issue
関連する他のissue

## 優先度
Low / Medium / High / Critical

## ステータス
Open / In Progress / Resolved / Closed

---
作成日: YYYY-MM-DD
最終更新: YYYY-MM-DD
```

## ステータスの定義

- **Open**: 未着手
- **In Progress**: 作業中
- **Resolved**: 実装完了、レビュー待ち
- **Closed**: 完全に解決済み

## 優先度の定義

- **Low**: 重要度が低い、時間があれば対応
- **Medium**: 通常の優先度、計画的に対応
- **High**: 高優先度、早急に対応が必要
- **Critical**: 最優先、即座に対応が必要

## 運用ガイドライン

1. **新しいissueの作成**: 技術的な課題や改善点を発見したら、新しいissueファイルを作成
2. **定期的なレビュー**: 週次でissueリストを確認し、優先度を更新
3. **完了時の更新**: issueが解決したら、ステータスを更新し、実装内容を記録
4. **クロスリファレンス**: 関連するissue同士をリンクして、全体像を把握しやすくする

## カテゴリ

今後、issueが増えてきたら、以下のようなカテゴリでフォルダ分けを検討:

- `ui-ux/`: UI/UX関連の課題
- `performance/`: パフォーマンス最適化
- `security/`: セキュリティ関連
- `backend/`: バックエンド関連
- `ios/`: iOSアプリ関連
- `api/`: API関連
