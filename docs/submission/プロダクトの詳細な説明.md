# プロダクトの詳細な説明

## プロダクト名

**すれちがいミュージック**

---

## コンセプト

> 「偶然の出会いが、新しい音楽と物語を生み出す」

街中ですれ違う見知らぬ人と、好きな音楽を自動で交換する体験を提供するモバイルアプリです。BLE（Bluetooth Low Energy）による近接検知を軸に、「すれ違い」という日常のひとときを、音楽発見の機会へと変えます。

さらに、すれ違った人々がそれぞれ歌詞の一節を投稿すると、AI が一つの楽曲を生成する「歌詞チェーン」機能により、見知らぬ人々の感性が交わって唯一無二の音楽が生まれる共創体験を実現します。

---

## 解決する課題

| 課題 | 本プロダクトの解決策 |
|------|-------------------|
| 音楽の趣味が偏りがち、新しい音楽に出会えない | 日常のすれ違いを通じた受動的な音楽発見 |
| SNS での音楽共有は能動的な行動が必要 | BLE による自動検知で意識せず交換が成立 |
| 音楽を介したリアルな人との繋がりが薄い | 場所・時間を共有した「縁」を記録・可視化 |

---

## 主要機能

### 1. すれ違い（BLE 近接検知）

BLE でスマートフォン同士が数メートル以内に近づくと、「すれ違い」が自動成立します。

- **プライバシー設計**: BLE 上にはサーバー発行の一時トークン（24時間有効、8バイト）のみを乗せ、ユーザー ID や個人情報は一切送信しない
- **誤検出防止**: RSSI フィルタ（-85dBm 未満は無視）、クライアント側 cooldown（5分）、debounce（30秒）を組み合わせて精度を高める
- **省電力設計**: Low Power モードでのアドバタイズ、バックグラウンド時は Opportunistic Scan（5秒スキャン・10秒スリープ）

すれ違い成立後、サーバーが各ユーザーのシェア楽曲・プレイリストを相互に公開し、プッシュ通知で知らせます。

### 2. 音楽共有・プレイリスト

- **楽曲連携**: Spotify / Apple Music の OAuth 連携により、実際に聴いている楽曲をシェア
- **プレイリスト**: ユーザーが独自のプレイリストを作成し、すれ違い相手に公開可能
- **楽曲検索**: Spotify API を使ったリアルタイム検索（最大50件、カーソルページネーション）

### 3. 歌詞チェーン（Lyric Chain）

本プロダクトの核となる AI 共創機能です。

```
① すれ違い成立
    ↓
② 歌詞の一節を投稿（任意・最大100文字）
    ↓
③ 4〜8人の歌詞が集まる（歌詞チェーン）
    ↓
④ Gemini 1.5 Flash がムード・ジャンル・テンポを分析し、タイトルを生成
    ↓
⑤ Google Lyria（lyria-002）が45秒の楽曲を自動生成
    ↓
⑥ 参加者全員にプッシュ通知で完成を伝える
```

- 複数ユーザーが同じ歌詞チェーンに参加する際の競合を、PostgreSQL の `SELECT FOR UPDATE` によるトランザクション内排他ロックで防止
- Gemini によるコンテンツモデレーション（投稿時にヘイト・暴力・成人向けコンテンツを自動検出・拒否）
- 生成失敗時は指数バックオフで最大3回リトライ

### 4. コメント・交流

- すれ違い相手に紐づいたコメント投稿（テキスト形式）
- すれ違い履歴の閲覧（カーソルページネーション）
- 楽曲へのいいね機能

### 5. 安全対策

| 機能 | 説明 |
|------|------|
| ブロック | ブロックしたユーザーとはすれ違い成立しない |
| ミュート | タイムラインに表示しない（相手には通知なし） |
| 通報 | ユーザー・コメントの2種類に対応 |
| Gemini モデレーション | 歌詞投稿時の自動チェック |

### 6. 通知

- すれ違い成立通知（バッチ処理：10〜20分ごとにまとめて送信し、連投防止）
- 歌詞チェーン完成通知
- Firebase Cloud Messaging（iOS: APNs、Android: FCM）

```
           |         |
           └────┬────┘
                ↓
   ┌────────────────────────────┐
   │         GCP Services       │
   │  Cloud SQL  Cloud Storage  │
   │  Vertex AI  FCM            │
   └────────────────────────────┘
```

### 認証フロー

```
[iOS] Sign in with Apple / [Android] Google Sign-In
    ↓
Firebase Auth（ID トークン発行）
    ↓
Bearer ヘッダ付きで Backend へ送信
    ↓
Firebase Admin SDK でトークン検証 → UID 取得
    ↓
Cloud SQL のユーザーレコードを参照・操作
```

### BLE すれ違いフロー

```
① サーバーから一時 BLE トークン取得（24時間有効）
② BLE アドバタイズ開始（APP_UUID + TOKEN_UUID）
③ スキャン（APP_UUID フィルタ）
④ TOKEN_UUID 抽出
⑤ RSSI チェック（< -85dBm は無視）
⑥ クライアント cooldown / debounce チェック
⑦ バックエンド API へ POST /encounters
⑧ サーバーがトークン → ユーザー ID 解決・重複チェック
⑨ Encounter 登録 → 通知 Outbox へ追加
```

## 特徴的な技術的工夫

### プライバシーファーストの BLE 設計

BLE 上に流れる情報はサーバー発行の一時トークン（16文字の hex 文字列）のみです。ユーザー ID・個人情報・楽曲情報は一切送信されず、トークンはサーバー側で即時失効可能な設計になっています。

### 楽曲生成パイプライン（Gemini + Lyria）

歌詞から直接 Lyria に送るのではなく、先に Gemini でムード・ジャンル・テンポを分析してから Lyria に渡すことで、歌詞の感情に合った楽曲を生成します。

```
歌詞チェーン（4〜8人分の歌詞）
    ↓
Gemini 1.5 Flash：ムード・ジャンル・テンポ推定 + タイトル生成
    ↓
Lyria（lyria-002）：45秒の楽曲生成
    ↓
Cloud Storage へアップロード → URL を DB に保存
    ↓
参加者全員へプッシュ通知
```

### Outbox パターンによる確実な非同期処理

通知送信・楽曲生成ジョブは DB トランザクション内で Outbox テーブルに書き込み、Worker が非同期で処理します。これにより、API レスポンス後にジョブが失われることなく、最低1回の実行を保証します。

### スキーマ駆動開発

Go コードのアノテーションから Swagger 2.0 スキーマを自動生成（swaggo）し、クライアントコード（openapi-generator）・API ドキュメントの自動生成を行うことで、モバイル・バックエンド間の認識ズレを防いでいます。

## デザイン・UX

### Urban Serendipity デザインコンセプト

「余白感を残しながら、アプリが生きている感覚を演出する」をテーマにした UI デザインです。

- **ドットグリッド背景**: 28pt 間隔・不透明度 15〜25% の微細なドットで「都市の構造」を表現
- **動的背景（DynamicBackground）**: アルバムアートから抽出した色彩による2層の Blob アニメーション
- **グラスモルフィズム（GlassmorphicCard）**: `.ultraThinMaterial` + ホワイトボーダー + シャドウで奥行きを表現
- **タイポグラフィ**: Black ウェイト（見出し）× Medium ウェイト（本文）の極端なコントラスト、等幅フォントによるデータ表示

### 主要画面

| 画面 | 役割 |
|------|------|
| オンボーディング | Apple/Google 認証 → プロフィール設定 → 権限許可（BLE・位置情報・通知） |
| ホーム | シェア中の楽曲 + 最近のすれ違いリスト + 今日の出会い件数 |
| すれ違い詳細 | アルバムアート大表示・ブラー背景・いいね（バウンスアニメ＋ハプティクス） |
| 歌詞チェーン | 参加者の歌詞一覧・生成楽曲の再生・チェーン進捗表示 |
| 検索 | Spotify 検索によるシェア楽曲・プレイリスト設定 |
| 設定 | プロフィール編集・音楽連携（Spotify / Apple Music）・ログアウト |
