# すれ違い音楽交換アプリ BLEアーキテクチャ設計

## 1. 目的

スマートフォン同士が近距離（数m〜数十m）ですれ違った際に、
ユーザーの「好きな曲」を自動で交換する。

BLEは **ユーザー識別のための一時ID交換のみ**に使用し、
実際のデータ取得はバックエンドAPIを利用する。

------------------------------------------------------------------------

# 2. 全体アーキテクチャ

BLE ↓ 軽いフィルタ（クライアント） ↓ API ↓ サーバー判定 ↓ 曲データ取得

役割を分離することで

-   BLE通信を最小化
-   セキュリティ確保
-   Android/iOS拡張性
-   サーバーでロジック制御

を実現する。

------------------------------------------------------------------------

# 3. システム構成

iOS / Android App │ │ BLE Advertising / Scan ▼ Ephemeral ID取得 │ ▼
クライアントフィルタ │ ▼ Backend API │ ▼ Encounter判定 │ ▼ Music Data
Fetch

# 4. BLEレイヤー

## 4.1 役割

BLEは以下のみを担当する。

-   近距離ユーザー検出
-   ephemeral_idの取得

BLEでは **ユーザー情報や曲情報は送らない。**

------------------------------------------------------------------------

## 4.2 BLE通信方式

advertise(ephemeral_id) scan

接続(connect)は使用しない。

理由

-   実装が簡単
-   電力消費が低い
-   スマホ同士でも安定
-   Android対応しやすい

------------------------------------------------------------------------

## 4.3 Advertising Payload

送信データ

ephemeral_id (8〜16 bytes)

格納場所

Manufacturer Data

------------------------------------------------------------------------

## 4.4 Ephemeral ID

ユーザー識別は **一時ID**を使用する。

理由

-   ユーザー追跡防止
-   プライバシー保護

例

ephemeral_id = hash(user_id + secret + time)

更新周期

10〜15分

------------------------------------------------------------------------

# 5. クライアント軽量フィルタ

BLEは同じユーザーを短時間で何度も検出する。

例

電車で隣 ↓ 30秒間 ↓ BLE検出10回

これを防ぐためクライアント側で軽量フィルタを行う。

------------------------------------------------------------------------

## 5.1 フィルタルール

同じephemeral_idは一定時間無視

例

cooldown: 10分

------------------------------------------------------------------------

## 5.2 クライアント処理

scan ↓ ephemeral_id取得 ↓ ローカルキャッシュ確認 ↓ 最近検出済みなら無視
↓ 未検出ならAPI送信

------------------------------------------------------------------------

# 6. APIレイヤー

BLEで取得したIDをサーバーへ送信する。

## API

POST /encounter

### Request

{ "self_user_id": "userA", "ephemeral_id": "abc123xyz" }

------------------------------------------------------------------------

# 7. サーバー処理

サーバーでは

ephemeral_id → user_id

を解決する。

------------------------------------------------------------------------

## 7.1 判定フロー

ephemeral_id受信 ↓ user_id解決 ↓ 既に今日交換済み？ ↓ YES → 無視 NO →
encounter登録 ↓ 曲データ取得

------------------------------------------------------------------------

# 8. Encounter制御

同じユーザーとの交換は

1日1回

に制限する。

------------------------------------------------------------------------

## DB構造例

### encounters

id user_a user_b encounter_date created_at

------------------------------------------------------------------------

## 判定SQL

SELECT \* FROM encounters WHERE user_a = ? AND user_b = ? AND
encounter_date = CURRENT_DATE

存在する場合

交換しない

------------------------------------------------------------------------

# 9. 曲情報取得

交換可能な場合

user_id ↓ favorite_song取得 ↓ クライアントへ返却

------------------------------------------------------------------------

# 10. セキュリティ設計

ユーザー追跡を防ぐため

user_idはBLEで送信しない

代わりに

ephemeral_id

を利用する。

------------------------------------------------------------------------

⑤ API送信

⑦ encounter判定
